THE THESIS

LOVE DESERVES REAL INFRASTRUCTURE

Dating platforms have APIs. Relationships don't.

The modern couple is surrounded by infrastructure — Tinder, WhatsApp, Instagram, Hinge, Discord — each optimized to extend the single life, not the shared one.

We think commitment deserves its own layer — not a dating app for couples, but a protocol underneath the ones that threaten it.

WHAT SURVEILLANCE GETS WRONG

Most "couples monitoring" apps are spyware with a love heart on the icon. One person installs. The other doesn't know. Data goes to a third party's server. Trust gets laundered through a database.

LoveClaw refuses that model. Both partners install. Both agree the rules. Signals stay on-device. The mesh is peer-to-peer — there is no server to subpoena, no vendor to breach, no dashboard for a stranger to read.

The cost of consent is higher. That's the point.

THE SEVEN PRINCIPLES

WE WILL NOT COMPROMISE ON THESE

Anything we build — the app, the protocol, the diary, the contract — has to pass all seven. If a feature fails one, we cut the feature, not the principle.

01 / 07
BOTH PARTNERS OR NEITHER

LoveClaw only works when both devices install it. One-sided monitoring is spyware. We refuse to ship it.

02 / 07
ON-DEVICE FIRST

Signals — GPS, apps, notifications, motion — stay on the phone they came from. Only scores and verdicts cross the mesh.

03 / 07
NO SERVER IN BETWEEN

Devices talk directly over AXL with Ed25519 keys. No LoveClaw backend holds your data. There is nothing to breach.

04 / 07
DETERMINISTIC OVER GENERATIVE

Breach detection is a rules engine — not an LLM. Claude writes the diary. Claude does not decide your relationship.

05 / 07
CONSENT IS THE KEY

Triggers are agreed together, before pairing. Nothing fires on a rule only one of you chose. The pact is symmetric.

06 / 07
OPEN SOURCE, AUDITABLE

Every rule, every weight, every message type — on GitHub. If we hide it, don't trust it. That's the rule for us too.

07 / 07
THE DISPUTE WINDOW

Every breach has 24 hours before execution. Humans review. The protocol serves the couple — not the other way around.

HOW IT WORKS

THREE LAYERS · ZERO MIDDLEMEN

The whole stack fits on two phones. A local agent reads signals. A mesh delivers them privately. An arbiter decides.

LoveClaw
Reads device signals on port :18789. Dating app installs, GPS drift, notification content, motion. Runs entirely local.

🌐
AXL MESH
P2P over port :9002. Ed25519 keys exchanged at pairing. End-to-end encrypted message types: handshake, breach, score, diary.

Deterministic rules fire at score ≥ 100. Evidence log preserved. 24h dispute window. Optional on-chain settlement.
📎 TECHNICAL FOOTNOTE
LoveClaw is a single HTML PWA. The entire app is ~1800 lines of vanilla JS. No build step, no framework, no backend. It runs on a Freedom Phone, an Android, or off a Pi in your kitchen. The whole stack is inspectable in an afternoon.

▸ WHAT LOVECLAW IS

  • A mutual pact between two consenting adults
  • A protocol that treats the relationship as the primitive
  • A signal layer that stays on the device where it started
  • A reward loop — the longer you're loyal, the richer your memory
  • A settlement option — on-chain stake, only if both parties want it
  • An open source manifesto with code attached

▸ WHAT LOVECLAW IS NOT

  • Spyware. One-sided install is technically impossible
  • A dating app. It's the layer beneath the dating apps
  • A cloud service. No backend. No central database. No server to hack
  • A marriage contract. Couples unpair anytime — the protocol knows no jurisdiction
  • An LLM judge. Breaches are rule-based. Claude only writes the diary
  • A surveillance tool. If it can be used to stalk, we've built it wrong
Trust is not a vibe.
It's an observable. A stream of signals both parties agreed to observe, scored against rules both parties agreed to sign.
If the stream holds, the bond holds.
If the rules fire, the pact names the breach itself.
